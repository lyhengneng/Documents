## Overview

OpenIM uses an **asynchronous message processing architecture** with Kafka as the central message queue. This ensures:

- ✅ **Reliability** - No message loss
- ✅ **Scalability** - Horizontal scaling possible
- ✅ **Performance** - Fast response times
- ✅ **Persistence** - Messages stored in MongoDB
- ✅ **Real-time** - WebSocket for online users

### Step 2: API → Message RPC

**Location:** `internal/rpc/msg/send.go:34-48`

The API calls the Message RPC service:

```go
func (m *msgServer) SendMsg(ctx context.Context, req *pbmsg.SendMsgReq) (*pbmsg.SendMsgResp, error) {
    // Encapsulate message data (add metadata)
    m.encapsulateMsgData(req.MsgData)

    // Route based on session type
    switch req.MsgData.SessionType {
    case constant.SingleChatType:
        return m.sendMsgSingleChat(ctx, req)
    case constant.NotificationChatType:
        return m.sendMsgNotification(ctx, req)
    case constant.ReadGroupChatType:
        return m.sendMsgGroupChat(ctx, req)
    default:
        return nil, errs.ErrArgs.WrapMsg("unknown sessionType")
    }
}
```

### Step 5: Message Transfer Service (Redis)

**Location:** `internal/msgtransfer/online_history_msg_handler.go`

The **toRedis** consumer processes messages:

```go
func (och *OnlineHistoryRedisConsumerHandler) ConsumeClaim(session, claim) {
    for msg := range claim.Messages() {
        // 1. Parse message from Kafka
        msgFromMQ := &sdkws.MsgData{}
        proto.Unmarshal(msg.Value, msgFromMQ)

        // 2. Store in Redis (fast access)
        och.handleChatWs2Mongo(ctx, msgFromMQ, ...)

        // 3. Forward to push topic
        och.toPushTopic(ctx, key, conversationID, messages)
    }
}
```

```go
func (c *ConsumerHandler) handleMs2PsChat(ctx context.Context, msg []byte) {
    // 1. Parse message
    msgFromMQ := &pbpush.PushMsgReq{}
    proto.Unmarshal(msg, &msgFromMQ)

    // 2. Route based on session type
    switch msgFromMQ.MsgData.SessionType {
    case constant.ReadGroupChatType:
        err = c.Push2Group(ctx, msgFromMQ.MsgData.GroupID, msgFromMQ.MsgData)
    default:
        err = c.Push2User(ctx, pushUserIDList, msgFromMQ.MsgData)
    }
}
```

### Step 8: Check Online Status

**Location:** `internal/push/push_handler.go:221-244`

```go
func (c *ConsumerHandler) GetConnsAndOnlinePush(ctx, msg, pushToUserIDs) {
    // Check who's online using Redis
    onlineUserIDs, offlineUserIDs, err := c.onlineCache.GetUsersOnline(ctx, pushToUserIDs)

    var result []*msggateway.SingleMsgToUserResults

    // Push to online users via WebSocket
    if len(onlineUserIDs) > 0 {
        result, err = c.onlinePusher.GetConnsAndOnlinePush(ctx, msg, onlineUserIDs)
        // → Sends to msggateway → WebSocket → Client
    }

    // Mark offline users for offline push
    for _, userID := range offlineUserIDs {
        result = append(result, &msggateway.SingleMsgToUserResults{
            UserID: userID,
            OnlinePush: false,  // Will trigger offline push
        })
    }

    return result
}
```

**Online Status Sources:**
1. Redis cache (fast lookup)
2. Message Gateway connections
3. User last activity

### Step 10: Offline Push

**Location:** `internal/push/push_handler.go:171-204`

For offline users:

```go
func (c *ConsumerHandler) Push2User(ctx, userIDs, msg) {
    // 1. Try online push first
    wsResults, err := c.GetConnsAndOnlinePush(ctx, msg, userIDs)

    // 2. If online push succeeded, skip offline push
    for _, v := range wsResults {
        if v.OnlinePush {
            return nil  // User got the message
        }
    }

    // 3. User is offline, send offline push
    if c.shouldPushOffline(ctx, msg) {
        err = c.offlinePushMsg(ctx, msg, needOfflinePushUserID)
        // → FCM, APNs, Huawei, Xiaomi, etc.
    }
}
```

**Offline Push Providers:**
- **FCM** (Firebase Cloud Messaging) - Android
- **APNs** (Apple Push Notification Service) - iOS
- **Huawei** - Huawei devices
- **Xiaomi** - Xiaomi devices
- **OPPO** - OPPO devices

- `push_handler.go` - Main push logic

## Timeline & Performance

### Request Timeline

| Step | Component | Time | Notes |
|------|-----------|------|-------|
| 1 | Client → API | 1-5ms | Network latency |

## Troubleshooting

### Message Not Delivered

**Check:**

1. **API Response:**
   ```bash
   grep "SendMsg" ./_output/logs/openim-service-log.YYYY-MM-DD
   ```

2. **Kafka Queue:**
   ```bash
   # Check Kafka topics
   kafka-console-consumer --bootstrap-server localhost:9092 --topic toRedis
   ```

3. **Redis Storage:**
   ```bash
   redis-cli
   > GET "msg:conversationID:seq"
   ```

4. **MongoDB Storage:**
   ```bash
   # Check MongoDB
   db.messages.findOne({serverMsgID: "msg-id"})
   ```

5. **Push Logs:**
   ```bash
   grep "push" ./_output/logs/openim-service-log.YYYY-MM-DD
   ```

### Common Issues

#### Issue: Message stuck in Kafka

**Solution:**
```bash
# Check consumer group lag
kafka-consumer-groups.sh --bootstrap-server localhost:9092 \
  --group msgToRedis --describe

# Restart consumers
mage restart
```

#### Issue: Online users not receiving messages

**Check:**
1. WebSocket connection status in Redis
2. MsgGateway logs
3. Network connectivity

#### Issue: Offline push not working

**Check:**
1. FCM/APNs credentials
2. Push notification tokens
3. Offline push configuration

## Summary

The OpenIM message sending flow is designed for:

✅ **Reliability** - Kafka ensures no message loss
✅ **Scalability** - Horizontal scaling via Kafka partitions
✅ **Performance** - Fast API response (10-35ms)
✅ **Real-time** - WebSocket for online users
✅ **Persistence** - MongoDB for long-term storage
✅ **Flexibility** - Webhooks for custom logic

The key insight is that **message sending is asynchronous** - the API responds quickly after queuing the message, while the actual delivery happens in the background through multiple services.
